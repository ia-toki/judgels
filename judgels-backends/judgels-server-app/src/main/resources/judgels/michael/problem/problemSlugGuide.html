Problem slug guide

General rules

  • A slug must be words separated by -. Each word must consist of lowercase letters or numbers.
  • Good examples: a-plus-b, tree-reconstruction, 2sat-construction.
  • Bad examples: a+b, tree--reconstruction, 2SAT-construction.

Past or upcoming contest problems

  • The slug should follow this format: <contest slug>-<problem slug>.
  • <contest slug> part:
    • It should identify the contest uniquely within the platform.
    • Similar contests should have the same format.
    • Good examples: ksn-2020, icpc-jakarta-2020, troc-22, toc-2018-feb.
    • Bad examples (considering the above): ksn, icpc-jakarta, troc, toc-2018.
  • <problem slug> part:
    • It should identify the problem uniquely within the contest.
    • It should consist of one or two key words taken from the problem title.
    • Good examples: mencari-bola, police, near.
    • Bad examples: a, b, pohon-merah-hitam, daratan-dan-es.

Modification of past contest problems

  • The slug should follow this format: <original slug>-modified[-<note>].
  • Examples:
    • troc-22-near-modified
    • troc-22-near-modified-functional (the grading engine is modified to Functional)

Unused problems (yet)

  • The slug should start with <your username>-.
  • If the problem is then used for a contest, the slug must be changed based on the above rules.
  • Example:dengklek-hello-world.

Problems that are just for testing

  • The slug should follow this format: <your username>-TESTING-<note>.
  • Please minimize the number of such problems.
  • Example:dengklek-TESTING-grading.

Problems that are broken / unused

  • The slug should follow this format: <your username>-UNUSED-<note>.
  • Example:dengklek-UNUSED-commit-error.
